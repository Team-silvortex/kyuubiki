Language packs and localization posture

Use this page when the question is how Kyuubiki should grow beyond the built-in interface languages without hard-wiring every future locale into the repo.

Current product posture

  • Built-in languages: English, Chinese, and Japanese remain first-class in the shipped UI.
  • Growth path: future languages should prefer language packs over hard-coded repo-wide copy forks.
  • Current delivery: packs are local-first today, imported as JSON through the Workbench system surface.

What a language pack is

A language pack is a partial override tree that patches the built-in Workbench copy dictionary. It does not need to redefine every key.

  • schema version: kyuubiki.language-pack/v1
  • required shape: identity fields plus an overrides object
  • recommended metadata: versionLine and targetAppVersion
  • fallback rule: missing keys keep using the built-in copy

Operator flow today

  • Download a template from System -> Config -> Language packs.
  • Edit the JSON locally for the target language.
  • Import the pack into the current browser workspace.
  • Check the displayed compatibility state before treating the pack as release-ready.
  • Export or remove the installed pack from the same UI surface.

What is intentionally not finished yet

  • No remote language-pack catalog download flow yet.
  • No signature or provenance chain yet.
  • No heavyweight browser-side schema validator dependency yet.

That is deliberate for now. The format is being stabilized first so remote catalog delivery can be added later without inventing a second translation contract.

Recommended direction

  • Keep core operator-critical copy shipped and reviewed in-tree.
  • Use packs for additional locales, regional variants, and faster iteration.
  • Route future remote pack delivery through the same visible System surface instead of a hidden auto-download path.
