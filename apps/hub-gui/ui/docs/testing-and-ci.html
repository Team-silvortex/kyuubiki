Main verification layers

  • Frontend test: typecheck plus production build validation.
  • Workflow preflight: workflow unit checks, topology regression, and browser-backed layout/search guards.
  • Integration smoke: orchestrator, agent, direct-mesh, and UI representative paths.
  • Desktop smoke: Hub, Installer, and Workbench shell checks.

When to run workflow preflight

  • Workflow builder layout changed.
  • Operator search, template-chain search, or workflow toolbar behavior changed.
  • Dataset editor, import diagnostics, snapshot, validation, or integrity panels changed.
  • A release includes workflow-heavy frontend work and should not rely on build success alone.

Operator flow

  • Start npm run dev inside apps/frontend.
  • Run the dedicated workflow preflight path from the central docs.
  • If it fails, inspect workflow layout/search behavior before chasing packaging or runtime causes.

Direct-mesh regression lane

The Docker direct-mesh benchmark now has its own checked-in baseline, compare path, and remote nightly wrapper instead of living only as an ad hoc lab command.

  • make test-integration-direct-mesh-docker runs the benchmark harness.
  • make test-integration-direct-mesh-docker-compare compares a summary against the baseline.
  • make test-integration-direct-mesh-docker-report runs a fresh benchmark and emits reports.
  • make test-integration-direct-mesh-docker-nightly drives the remote kyuubiki-lab regression wrapper.
  • The container defaults to DOCKER_RUN_NETWORK=host so LAN agents remain discoverable.
