Component Integrity · Protocol

kyuubiki.component-integrity/v1

Component integrity is the long-term protocol for declaring what a Kyuubiki component owns, requires, protects, and may clean. It turns installation integrity from a one-time repair process into a contract that every new component must join.

Why this exists

Kyuubiki has many installable and runtime-adjacent parts: desktop shells, Installer logic, runtime state, release snapshots, brand metadata, future agents, operator libraries, and store sources. A component may add files, but it must also declare ownership and safety behavior. Otherwise repair and update tooling cannot know whether a path is required, user-owned, protected, or removable residue.

Component fields

  • id: stable component id, such as desktop.hub or runtime.state.
  • kind: class of component, such as desktop-shell, installer, or brand-metadata.
  • version: expected product-line version for the component.
  • owner: human/tooling ownership hint used by reports.
  • owned_paths: path roots the component is allowed to claim.
  • required_paths: paths that must exist for the component to be healthy.
  • protected_paths: paths repair must not remove and that must also appear in the global protected set.
  • removable_patterns: component-local residue patterns, still constrained by global repair policy.
  • visible_rules: operator-facing read-only or editable behavior metadata.

Current components

  • workspace.root: root sentinel for required-layout coverage.
  • brand.shared: shared brand metadata, including releaseVersion.
  • installer.core: installer contract, staging, and runtime payload roots.
  • desktop.installer: Installer desktop shell and brand metadata.
  • desktop.hub: Hub desktop shell and brand metadata.
  • desktop.workbench: Workbench desktop shell and brand metadata.
  • release.snapshots: protected lightweight version-history snapshots.
  • runtime.state: runtime state roots and stale process-file cleanup.

Validation rules

  • Every required installation path must be covered by at least one component's owned_paths.
  • Every component required_paths entry must sit inside one of that component's owned paths.
  • Every component protected_paths entry must also be present in the global protected path set.
  • Installer integrity reports render required_path_coverage; healthy current output is 15/15.
  • Brand metadata is now a required component concern, so desktop shells must expose matching releaseVersion.

Adding a new component

  1. Add the component's required paths to required_layout if they must exist after install or repair.
  2. Add a components[] entry in deploy/installation-integrity-contract.json.
  3. Keep every required path inside the component's owned_paths.
  4. Add protected paths globally before declaring them in the component.
  5. Run cargo test -q -p kyuubiki-installer and cargo run -q -p kyuubiki-installer -- installation-integrity.
