Runtime and deployment modes

The product is intentionally deployable in multiple shapes, but those shapes must stay explicit and understandable so operators and runtime engineers know which contracts apply.

Back to book Previous chapter Next chapter

Deployment shapes

  • local workstation: frontend + orchestrator + local Rust agents.
  • cloud control plane: browser and orchestrator with remote persistence and external agents.
  • distributed control plane: separated control plane with remotely deployed solver nodes.
  • headless peer mesh: future or selective mode where Rust agents can cluster without GUI or Phoenix on the hot path.

Operational goal

Each deployment mode should remain legible in terms of storage, discovery, authority, and update behavior. Documentation should make those tradeoffs visible instead of hiding them in environment files.

Remote deployment maturity

Installer remote deployment is currently a policy-bounded SSH pilot, not yet a full production deployment service. The roadmap is to move toward explicit deployment plans, durable journals, artifact delivery, host trust, and physical/containerized remote tests. The first structured plan preview now lives in the Installer CLI as remote-deployment-plan, with matching journal preview records available through remote-deployment-journal.
