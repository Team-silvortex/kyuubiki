Rust solver optimization pack

Retained matrix optimizations for thermal quad benchmarks

This page records the matrix-side optimization set that is currently worth keeping in the Rust solver, the experiments that were rejected, and the benchmark evidence that justifies the retained pack.

Keep these changes

  • Heat quad precompute split: use precompute_heat_plane_triangle_element_from_nodes so quad precompute stops cloning full node lists.
  • Stage RSS visibility: keep profile_heat_plane_quad_2d and the stage chain precompute -> assemble_global -> reduce_system -> solve_system -> assemble.
  • Row-capacity hints: keep SparseMatrix::with_uniform_row_capacity and average_row_non_zero_hint for derived sparse matrices.
  • Sorted append for derived matrices: keep push_sorted_entry for reduced and scaled matrices that are already emitted in non-decreasing column order.
  • Reduced and scaled matrix paths: keep the sorted-append path in reduce_sparse_system, reduce_sparse_system_with_prescribed, and scale_sparse_matrix.
  • Solver success-path lifetime trim: keep the change that avoids holding one extra long-lived scaled sparse matrix after compression.

Rejected experiments

  • Unordered global assembly then finalize: structurally interesting, but it did not beat the retained variant and should not come back without new evidence.
  • Per-iteration diagonal reread in CG: regressed local benchmark behavior.
  • Using stage-end RSS as peak-RSS proof: useful for steady-state interpretation, but not enough to explain release-mode spikes on its own.

Current evidence

  • Local optimized worktree: heat-plane-quad-10k repeat 3, median 657.5775 ms, peak RSS 15 MiB.
  • Clean HEAD control group: same command, median 6797.5630 ms, peak RSS 22 MiB.
  • Physical lab release reference: thermal-core repeat 3 on kyuubiki-lab, median 30.8742 ms, peak RSS 29 MiB.
  • Stage-end RSS on the lab: stable at 15 MiB, which suggests the remaining release spike is a short-lived internal allocation rather than a stage-end steady-state footprint.

How to evaluate future changes

  1. Run cargo test -p kyuubiki-solver.
  2. Run the local thermal benchmark: cargo run -p kyuubiki-benchmark -- --profile 10k --matrix thermal --case heat-plane-quad --repeat 3.
  3. If the local result is credible, run the physical-lab release benchmark against the checked baseline.
  4. Only keep a change if it improves runtime, memory, or clarity without introducing a benchmark regression.

Next likely targets

  • Short-lived allocation spikes inside compressed-matrix solve internals.
  • Compressed representation construction costs beyond the current sorted-append and row-capacity improvements.
  • Extending the same retained derived-matrix strategy to other solver families only after family-specific benchmarks exist.
