Chapter 1: What Kyuubiki is

Kyuubiki is an engine-first FEM workstation and control plane. The user-facing surface is split across desktop shells and a browser workbench, while the execution layer is split across a Phoenix control plane and Rust solver agents.

  • Hub: operator shell, runtime control, workload shelf, docs, packaging entrypoint.
  • Workbench: modeling, analysis setup, results, and future deeper 3D interaction.
  • Installer: bootstrap, deployment prep, installation integrity, release structure.
  • Orchestrator: job control, persistence, workflow coordination, remote/runtime management.
  • Rust agents: solver data plane, headless execution, future mesh and peer-networking growth.

Chapter 2: What tamamono 1.x is optimizing for

The current line is not supposed to grow mainly by feature count. It is supposed to harden trust: numerical accuracy, stable contracts, cleaner boundaries, safer deployment, and a more coherent operator experience.

  • Keep major version posture conservative.
  • Prefer verified solver growth over wide speculative UI growth.
  • Make documentation, automation, and release behavior explicit instead of folklore-driven.

Chapter 3: The hard architecture boundaries

Kyuubiki depends on preserving certain splits. Frontend, runtime, orchestrator, and agents are not supposed to blur into one another. The Hub is not the orchestrator. The Workbench is not the runtime. The agent is not the UI.

  • Hub manages entry, operator flow, runtime switching, and packaging posture.
  • Workbench owns the concrete analysis workflow and user-facing authoring surface.
  • Installer owns deployment structure and installation policy.
  • Agents remain runtime-side, headless, and Rust-first.
  • One orchestrator controls an agent at a time unless the system is intentionally in offline/decentralized mesh mode.

Chapter 4: Runtime and deployment modes

The product is intentionally deployable in multiple shapes, but those shapes must stay explicit and understandable.

  • local workstation: frontend + orchestrator + local Rust agents.
  • cloud control plane: browser and orchestrator with remote persistence and external agents.
  • distributed control plane: separated control plane with remotely deployed solver nodes.
  • headless peer mesh: future or selective mode where Rust agents can cluster without GUI/Phoenix on the hot path.

Chapter 5: Workflow and operator philosophy

Kyuubiki is moving beyond isolated single operators. The workflow direction is shader-like or neural-network-like composition: operators connected through a stable cross-operator data contract.

  • Workflow graph growth is intentional, not ad hoc.
  • Dataset transport should be portable and contract-driven, with an ONNX-like spirit.
  • Operator library ownership remains centralized instead of fully replicated to every agent.
Open chapter page workflow-graph.md workflow-dataset.md operator-library-centralization.md

Chapter 6: SDK and automation surfaces

There are multiple extension surfaces, and they must not be confused.

  • Rust operator SDK: for authoring operator capabilities and extending the solver/runtime side.
  • Headless SDKs: for automation, agents, external tools, and AI clients running outside the frontend.
  • WASM Python UI automation: frontend-owned and stability-sensitive, with product-owned selector ids instead of localized copy.
Open chapter page operator-sdk.md headless-sdks.md ui-automation-contract.html headless-agent-contract.md

Chapter 7: Accuracy, trust, and release safety

Solver trust is a first-class product concern. So are safety, installation integrity, update visibility, and release repeatability.

  • Accuracy baselines and expansion plan are explicit.
  • Security and destructive actions are meant to stay visible and auditable.
  • Installation and update behavior must be configuration-visible and residue-conscious.
  • Workflow-heavy frontend changes should pass a dedicated preflight instead of relying on build success alone.
