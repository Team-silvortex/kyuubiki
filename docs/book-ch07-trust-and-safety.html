Accuracy, trust, and release safety

Solver trust is a first-class product concern. So are safety, installation integrity, update visibility, and release repeatability. The product should be understandable and auditable, not magical.

Back to book Previous chapter Next chapter

Trust pillars

  • Accuracy baselines and expansion plans are explicit.
  • Security and destructive actions stay visible and auditable.
  • Installation and update behavior are configuration-visible and residue-conscious.
  • Credentials live in Kyuubiki-owned sandbox roots, not hidden system-global folders or ordinary project config.
  • Mobile credential access is handle-based through native secure storage; WebView code must not receive raw private keys or long-lived token bytes.
  • Component integrity is a protocol: new components must declare ownership, required paths, protected paths, cleanup scope, and visible rules.

Release expectation

A stronger product line is one where operators can inspect behavior, engineers can verify behavior, and release owners can repeat behavior.
