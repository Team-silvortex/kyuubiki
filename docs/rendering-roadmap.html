Rendering Roadmap

tamamono 1.x render capability upgrade path

This page turns the current rendering reality into a concrete staged plan. The goal is not to chase visual polish first. The goal is to let advanced multi-operator simulation features stay truthful, scalable, and eventually viewport-native.

Current line: tamamono 1.13.x North star: FEM-side Blender viewport Status: planning contract

assessment

Current rendering posture

The current workbench renderer is already good enough for operator growth, workflow debugging, and moderate result inspection. It is not yet a true simulation viewport engine. Most visible rendering still runs through SVG-driven view components and progressive DOM budgeting rather than a dedicated field renderer.

  • strong today: workflow editing, scalar-field selection, result summaries, moderate 2D and truss inspection
  • acceptable with caution: larger studies that rely on progressive batch reveal instead of full interactive density
  • not ready yet: dense plane-field clouds, layered electrostatic plus thermal plus thermo overlays, smooth deform-plus-field playback, Blender-class viewport feel
evidence

Why this assessment is concrete

  • viewport dispatch: apps/frontend/src/components/workbench/workbench-viewport.tsx selects SVG-oriented renderers for axial, line, plane, and truss3d views
  • render strategy: progressive reveal is handled by requestAnimationFrame plus renderBatchSize instead of a persistent GPU scene graph
  • field surface: result fields are already broad in workbench-viewport-core.ts, which means the data model is moving faster than the viewport substrate
  • result chunking: chunkDirectMeshResult in apps/frontend/src/lib/direct-mesh/results.ts shows that large-result access is already becoming a concern
rule

Decision rule for the next minors

Keep shipping operator and workflow capability now, but do not pretend the current SVG viewport is the final answer for high-density simulation visuals. New advanced physics features may land before the final render substrate does, as long as they remain inspectable, testable, and performance-bounded.

1.6.x

Stabilize the current viewport contract

Finish the separation between simulation truth and visual representation. This is the last stage where SVG remains the primary renderer for most result views.

Must land

  • result surface contract: define which result fields are viewable, selectable, exportable, and benchmarked per study family
  • render diagnostics: expose node count, element count, render-window limits, and progressive-render state in the workbench UI
  • field fallback policy: when density is too high, fall back to summary, hotspot, slice, or sampled views instead of freezing the viewport
  • selection integrity: keep picking, node focus, element focus, and workflow trace links stable even under partial render budgets

Must not attempt yet

  • full viewport rewrite
  • unbounded multi-layer overlay rendering
  • cinematic animation polish before field correctness
1.7.x

Introduce a dedicated plane-field renderer

This is the real turning point. The highest pressure area is not the whole workbench at once. It is 2D plane-field rendering for dense scalar and vector results.

Must land

  • new render lane: add a dedicated plane-result subrenderer on Canvas, WebGL, or WebGPU, without rewriting all workbench chrome
  • triangle and quad support: same field view path must support heat, electrostatic, structural plane, and thermo plane studies
  • palette contract: unify contour palette, clamps, outlier handling, legend generation, and no-data behavior
  • deformation composition: allow deform-only, field-only, and deform-plus-field modes without duplicating simulation transforms
  • render benchmarks: establish practical targets for small, medium, and stress-case plane meshes

Success condition

Advanced plane operators become believable because the viewport can display dense fields directly instead of reducing everything to sampled SVG shapes.

1.13.x

Make multi-physics visualization composable

Once plane-field rendering is real, the next step is controlled layering. This is where electrostatic-to-heat-to-thermo chains start feeling like a native simulation language instead of several separate reports.

Must land

  • layer stack: explicit visibility and order for deformation, scalar field, vector glyphs, constraints, loads, selection, and warnings
  • workflow-linked views: clicking a workflow node can drive the active result layer without rebuilding the whole viewport state
  • cross-operator comparison: let users compare source field, bridged field, and downstream response inside one visual session
  • snapshot discipline: viewport state snapshots must stay bounded in memory and intelligible in audit exports
1.13.x+

Converge toward a true FEM creation viewport

Only after the field renderer and layer system exist should the project lean into heavier viewport ambitions.

  • space-view maturity: improve 3D structural rendering, clipping, and density-aware picking
  • material and operator previews: shader-like previews for workflow templates, operator packs, and reusable composite assets
  • interactive analysis modes: probes, cuts, path plots, temporal comparisons, and guided diagnosis surfaces
scope

What can ship now without waiting

  • operators: new electrostatic, heat, thermal, and transform operators
  • bridge contracts: richer cross-operator field transfer rules and reduction policies
  • workflow features: graph insertion, composite templates, run traces, benchmark hooks, audit exports
  • SDK growth: headless workflow and operator execution contracts

These stay valid because the rendering layer is a consumer of simulation truth, not the truth itself.

guardrails

What would be a mistake right now

  • mistake 1: treating SVG batching as a permanent high-density rendering solution
  • mistake 2: coupling workflow semantics to whatever the current viewport can easily draw
  • mistake 3: shipping heavy visual layering before a memory and snapshot budget exists
  • mistake 4: benchmarking only solver time while ignoring result-surface cost
benchmarks

Render metrics that should become first-class

  • time to first view: time until a user sees a meaningful result surface
  • time to full detail: time until all intended layers are available
  • interactive floor: pointer and selection responsiveness while the viewport is busy
  • memory ceiling: viewport plus snapshot memory under repeated workflow runs
  • degrade quality: whether fallback modes stay honest instead of silently dropping important context
bottom line

Simple project stance

The current renderer is enough to support serious operator progress. It is not enough to be the final host for the most advanced simulation features. So the correct strategy is not to pause feature work. The correct strategy is to keep advancing operators while preparing a dedicated field-render lane in the next minor line.
